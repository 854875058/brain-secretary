# 需求文档（Brain Secretary）

> 创建：2026-03-06
> 维护者：你 + 本仓库协作的 AI（Codex / OpenClaw）
> 最近校准：2026-03-16
> 目标：把需求“写死”，避免口口相传导致跑偏

---

## 1. 项目目标（What）

把 QQ 私聊 / 群聊里的自然语言指令，交给 OpenClaw 上运行的“大脑”去理解、记忆、分发、验收，再把结果通过 QQ 回复给你。

你关心的是：

- 稳定
- 可持续迭代
- 具备记忆与任务闭环
- 能运维、能排障、能恢复

---

## 2. 当前落地基线（Now）

截至 2026-03-16，当前已落地的系统基线是：

- 主 QQ 入口已经切换到 OpenClaw 原生 `qqbot` 渠道
- 当前正式主链路是：`qqbot/default -> qq-main -> 子 agents -> qq-main -> QQ`
- 当前真实多 agent 角色已经固定：
  - `qq-main`：协调大脑
  - `brain-secretary-dev`：工程实施子 agent
  - `brain-secretary-review`：方案 / 验收子 agent
  - `auto-evolve-main`：自动进化专用内部协调 agent
- `qq-bot/` 不再承担现网主入口职责，仅保留为历史参考和辅助多 QQ 桥接层
- Paperclip 已接到系统后方，承担协同投影 / 网页任务控制面，不替代主 QQ 入口

---

## 3. 核心需求（Must Have）

### 3.1 OpenClaw 作为主脑

- QQ 的核心交互默认走 OpenClaw，而不是直接调用第三方 Chat Completions
- 系统要能在同一会话里持续对话，保持上下文连续
- 大脑需要负责理解意图、拆任务、委派执行、验收结果、统一回复

### 3.2 记忆与上下文

至少保留两层记忆：

1. 会话记忆：QQ 对话需要稳定映射到 OpenClaw 会话，保证多轮上下文连续
2. 长期记忆：项目路径、规则、偏好、待办、关键决策要沉淀到 `MEMORY.md` / `memory/*.md`

补充要求：

- 自动进化默认使用 fresh session，避免历史上下文污染
- 长期稳定事实应优先写入仓库文档和记忆文件，而不是只留在聊天里

### 3.3 真实多 agent 协同

- 不能用“单 agent 假装多角色”替代真实子 agent 协同
- `qq-main` 必须保持总协调角色，不直接退化成某一个子 agent
- 工程实施、验收复核、自动进化协调应有明确角色边界

### 3.4 运维统一入口

- 涉及部署、启停、状态、日志、端口检查，优先使用 `scripts/ops_manager.py`
- 运维真源固定为 `ops/deployment_manifest.json`
- 如果部署方式、服务名、端口、agent id、workspace、模型发生变化，必须同步更新相关文档

---

## 4. 非功能需求（Should Have）

- 可观测性：日志清晰，状态和错误能快速定位
- 安全边界：默认不应无限开放陌生来源；需要明确认证和限流策略
- 可恢复性：OpenClaw、模型代理、网关、反向代理异常时要能快速自检和恢复
- 可扩展性：后续可以继续接更多项目仓库、更多子 agent、更多辅助入口
- 文档一致性：不能让旧方案文档长期冒充当前实现

---

## 5. 当前架构约束（Constraints）

- `qqbot/default` 必须绑定 `qq-main`，不能直接绑定到子 agent
- `auto-evolve-main` 只给自动进化守护使用，不能承接主 QQ 入口
- 不要为临时验证重新启用已退役的 `openclaw-qq-bridge.service` 和 `napcat-qq.service`，除非明确回滚
- `qq-bot/` 现在不是现网主入口，相关逻辑按历史兼容和辅助桥接理解
- Paperclip 是后方控制面，不是新的 QQ 主入口
