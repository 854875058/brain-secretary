## Slide 3

### 标题

什么是 Spec Coding

### 屏幕提示

- 目标
- 约束
- 验证
- 输出

### 口播

我现在对 Spec Coding 的理解，不是“把 Prompt 写长一点”，也不是“让 AI 多看几段描述”。

它的本质，是先把一个任务说清楚，再让 AI 去执行。

这个“说清楚”至少要包括四件事：

第一，目标是什么。  
第二，边界和约束是什么。  
第三，验证方式是什么。  
第四，最终输出是什么格式。

所以在我这里，Spec 不是一份漂亮文档，它更像是一份任务合同。

先把合同写清楚，再让 AI 开工。

## Slide 4

### 标题

为什么单 Agent 不够

### 屏幕提示

- 既理解需求，又写实现，又自己验收
- 容易理解偏
- 容易越边界
- 容易没验证就说完成

### 口播

很多人用 AI 编程的时候，默认是一个 Agent 从头做到尾。

但真实项目里，单 Agent 最大的问题是，它经常同时扮演三个角色：  
它既理解需求，又做实现，又自己宣布完成。

这样会带来几个很常见的问题。

第一，需求理解偏了。  
第二，改动越边界了。  
第三，没有验证，就直接说自己做完了。

所以在 Demo 场景里，一个 AI 够用；但在工程交付场景里，单 Agent 往往不够稳。

## Slide 5

### 标题

三节点 Agent 架构

### 屏幕提示

- `qq-main`：协调节点
- `brain-secretary-dev`：实施节点
- `brain-secretary-review`：验收节点
- `auto-evolve-main`：自动巡检与自动闭环扩展

### 口播

所以我们在这个项目里，做了一次角色拆分。

第一个节点是 `qq-main`。  
它更像项目经理加技术负责人，负责理解需求、拆任务、控边界、最后汇总结果。

第二个节点是 `brain-secretary-dev`。  
它像开发工程师，专门负责工程实施，比如改代码、改脚本、跑验证。

第三个节点是 `brain-secretary-review`。  
它像 reviewer 或者 QA，不负责产出实现，而是专门负责验收、挑错、补风险、提返工意见。

另外我们还有一个 `auto-evolve-main`，它不承接 QQ 主入口，而是用来做自动巡检和自动闭环，这是这套架构从“有人发需求”扩展到“无人值守运行”的关键。

## Slide 6

### 标题

Spec 在三节点之间如何流转

### 屏幕提示

需求 -> 协调节点 -> 实施节点 -> 验收节点 -> 汇总结果/异常 -> 人

### 口播

在这个结构里，Spec 的意义也发生了变化。

它不再只是“写给一个 AI 看的一段提示词”，而是三个节点共享的任务契约。

协调节点拿到需求之后，先把目标、约束、验证方式和输出格式整理出来。  
实施节点按这份合同去做实现。  
验收节点也用这份合同来判断，这个任务到底算不算完成。

最后，系统再把结果或者异常汇总给人。

所以这条链路的核心，不是多开几个 AI，而是让“理解、实施、验收”真正分离。

## Slide 7

### 标题

项目是什么

### 屏幕提示

- OpenClaw 多 Agent 工程系统
- QQ 入口统一接到 `qq-main`
- `dev/review` 承担实施与验收
- Paperclip 负责协同投影和可视化

### 口播

我们当前这个项目，本质上是一个围绕 OpenClaw、多 Agent、QQ 入口和工程闭环搭起来的系统。

QQ 侧的统一入口接到 `qq-main`。  
它收到需求之后，可以把任务派给 `brain-secretary-dev` 和 `brain-secretary-review`。  
如果是自动巡检或自动进化场景，则由 `auto-evolve-main` 来驱动。

同时，整个协同过程还可以投影到 Paperclip，用来做过程可视化和状态观察。

所以它不是一个单纯的聊天机器人项目，而是一个面向工程交付的多 Agent 系统。

## Slide 8

### 标题

项目演进路线

### 屏幕提示

- 阶段一：设计约束与角色拆分
- 阶段二：项目搭建与基础链路打通
- 阶段三：功能开发与多 Agent 协同
- 阶段四：部署、验收与自动闭环

### 口播

如果把项目演进过程压缩来看，大概可以分成四个阶段。

第一阶段，是设计约束和角色拆分。  
也就是先想清楚，哪些角色要拆，哪些边界要守。

第二阶段，是基础链路打通。  
包括入口、路由、配置、运维脚本这些工程基础。

第三阶段，是功能开发。  
这里重点不是功能数量，而是多 Agent 协同开始真正发生。

第四阶段，是部署、验收和自动闭环。  
也就是从“能做事”进一步走到“能稳定交付”。

如果你后面需要讲你自己的 10 天时间线、多少条指令、多少轮演进，这一页可以替换成你自己的真实数据。

## Slide 10

### 标题

工程化落地：不是聊天，而是系统

### 屏幕提示

- `CLAUDE.md` / Rules：边界与约束
- `ops_manager.py`：统一运维入口
- `deployment_manifest.json`：部署真源
- 结构化报告与异常汇总：只看异常

### 口播

这套方法真正有价值的地方，不在于“会聊天”，而在于它开始有工程化约束。

我们用规则文件约束边界。  
用统一运维入口管理服务。  
用部署清单统一维护真源。  
用结构化报告收敛结果。  
再用异常汇总把真正需要人关注的问题单独拎出来。

也就是说，AI 不是干完就算，而是必须留下可验证、可追责、可复核的结果。

## Slide 11

### 标题

开发者角色的重构

### 屏幕提示

- 从“亲手执行每一步”
- 变成“定义目标、设规则、看异常、做取舍”

### 口播

我觉得这套实践对我最大的改变，不是代码写得更快了，而是开发者角色变了。

以前开发者更多像执行者。  
现在更像一个系统设计者和监督者。

你要做的，不再只是亲手写每一行代码，而是定义目标、设规则、看异常、做取舍。

执行本身，越来越可以交给 AI。  
但边界、验收、风险判断，仍然需要人来负责。

这才是我理解的 Spec Coding 和多 Agent 最大的价值。
